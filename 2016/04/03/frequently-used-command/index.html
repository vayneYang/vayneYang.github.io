单引号，双引号，反引号

  • 首先，单引号和双引号，都是为了解决中间有空格的问题。单引号将剥夺其中的所有字符的特殊含义，而双引号中的‘$’（参数替换）和‘`’（命令替换）是例外。
  • 反引号：在执行一条命令时，会先将其中的 ``，或 $() 中的语句当作命令执行一遍，再将结果加入到原命令中重新执行。

五个查找命令：find, locate, whereis, which, type

  1. find: 最常见和最强大的查找命令，你可以用它找到任何你想找的文件。
  2. locate命令其实是“find -name”的另一种写法，但是要比后者快得多，原因在于它不搜索具体目录，而是搜索一个数据库（/var/lib/locatedb），这个数据库中含有本地所有文件信息。
  3. whereis，只能用于程序名的搜索，而且只搜索二进制文件（参数-b）、man说明文件（参数-m）和源代码文件（参数-s）。如果省略参数，则返回所有信息。
  4. which，在PATH变量指定的路径中，搜索某个系统命令的位置，并且返回第一个搜索结果。
  5. type，type命令其实不能算查找命令，它是用来区分某个命令到底是由shell自带的，还是由shell外部的独立二进制文件提供的。如果一个命令是外部命令，那么使用-p参数，会显示该命令的路径，相当于which命令。

nslookup: nslookup简介

dig简单命令格式：
dig @server name type

  • querytype：可设置A/AAAA/PTR/MX/ANY等值，默认是查询A记录。
    • A：查看主机的IPv4地址。A记录，也称为主机记录，是DNS名称到IP地址的映射，用于正向解析。
    • AAAA：查看主机的IPv6地址
    • ANY：查看关于主机域的所有信息
    • CNAME：查找与别名对应的正式名字。CNAME记录，也是别名记录，用于定义A记录的别名。
    • HINFO：查找主机的CPU与操作系统类型
    • MINFO：查找邮箱信息
    • MX：查找邮件交换信息。邮件交换器记录，用于告知邮件服务器进程将邮件发送到指定的另一台邮件服务器。（该服务器知道如何将邮件传送到最终目的地）。
    • NS：查找主机域的域名服务器。NS记录，用于标识区域的DNS服务器，即是说负责此DNS区域的权威名称服务器，用哪一台DNS服务器来解析该区域。一个区域有可能有多条ns记录，例如zz.com有可能有一个主服务器和多个辅助服务器。指出当前区域内有几个DNS服务器在提供服务
    • PTR：查找与给定IP地址匹配的主机名。是IP地址到DNS名称的映射，用于反向解析。
    • RP：查找域负责人记录
    • SOA：查找域内的SOA地址。用于一个区域的开始，SOA记录后的所有信息均是用于控制这个区域的，每个区域数据库文件都必须包谷一个SOA记录，并且必须是其中的第一个资源记录，用以标识DNS服务器管理的起始位置，SOA说明能解析这个区域的dns服务器中哪个是主服务器。指明主DNS服务器
    • UINFO：查找用户信息
  • -c选项，可以设置协议类型（class），包括IN(默认)、CH和HS。
  • -f选项，dig支持从一个文件里读取内容进行批量查询 dig -f querylistfile -c IN -t A
  • -4和-6两个选项，用于设置仅适用哪一种作为查询包传输协议，分别对应着IPv4和IPv6
  • -t选项，用来设置查询类型，默认情况下是A，也可以设置MX等类型
  • -q选项，其实它本身是一个多余的选项，但是它在复杂的dig命令中又是那么的有用。-q选项可以显式设置你要查询的域名，这样可以避免和其他众多的参数、选项相混淆，提高了命令的可读性
  • -x选项，是逆向查询选项。可以查询IP地址到域名的映射关系

re

>>> m = re.match(r"(\w+) (\w+)", "Isaac Newton, physicist")
>>> m.group(0)       # The entire match
'Isaac Newton'
>>> m.group(1)       # The first parenthesized subgroup.
'Isaac'
>>> m.group(2)       # The second parenthesized subgroup.
'Newton'
>>> m.group(1, 2)    # Multiple arguments give us a tuple.
('Isaac', 'Newton')

>>> m = re.match(r"(\d+)\.(\d+)", "24.1632")
>>> m.groups()
('24', '1632')

mongo

Cursor 超时

pymongo.errors.CursorNotFound: Cursor not found, cursor id: 581302468884

原因：db.collection.find() 的时候，它返回的不是所有的数据，而实际上是一个cursor。 10 分钟无操作后这个 cursor 即超时。

  • 解决一：
    控制batch的大小，使服务器返回的文档在10分钟内一定能处理完。
    db.collection.find().batch_size(30) # batch为文档数

  • 解决二：
    [item for item in db.collection.find()] # 一次取完所有文档存为list再做处理

查询

db.inventory.find( { field: { $exists: true, $nin: [ 5, 15 ]}})  //查找field字段存在且不为5或15。

find({ "field" : { $gt: value }})  //查询大于某数的记录。

find() 结果乱序

mongodb 不保证文档顺序，当对 collection 做了一次修改并且修改后的这条数据大于原来的大小，那么这条文档就会被扔到 collection 的最后。这就导致下次 find() 得到的文档失序。

因此我选择了sort:

db.collection.find().sort('_id', ASCENDING)

但是效率低了一半。。再找找有没有更好的解决办法。索引？试试看。
